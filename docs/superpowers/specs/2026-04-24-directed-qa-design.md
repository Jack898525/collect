# 定向问答设计说明

日期：2026-04-24

## 1. 目标

在现有“自由 QA 收集平台”中新增一个独立模块“定向问答”。

该模块允许管理员预先录入一批问题，学生围绕这些预设问题提交回答。定向问答的回答与现有自由 QA 分开展示，但其被采纳后的得分需要并入现有排行榜。

## 2. 范围

本次实现包含：

- 前台新增“定向问答”模块
- 后台新增定向题目管理
- 后台新增定向回答管理
- 定向回答采纳计分并并入总排行榜
- 本地启动站点供人工检查

本次实现不包含：

- 后台鉴权方案重构
- 现有自由 QA 展示墙改版
- 新增完整自动化测试体系

## 3. 业务规则

### 3.1 定向题目

- 题目由管理员在后台创建
- 题目至少包含题干文本
- 题目支持启用和停用
- 前台仅展示处于启用状态的题目

### 3.2 定向回答

- 学生复用现有身份信息字段：姓名、学号、昵称
- 学生可以对同一题目重复提交多次回答
- 每条回答独立进入后台审核
- 回答默认公开可见，除非管理员隐藏
- 定向回答只在“定向问答”模块内展示，不混入现有 QA 展示墙

### 3.3 采纳与计分

- 定向回答被采纳后，计 1 分
- 排行榜分数为：
  - 自由 QA 中被采纳的数量
  - 加上定向问答中被采纳的数量
- 同一学生在同一定向题目下，最多只能保留 1 条已采纳回答
- 当管理员采纳该学生在该题下的新回答时，系统必须自动取消该学生在该题下其他已采纳回答
- 同一学生在不同题目上的已采纳回答互不影响

## 4. 数据模型

推荐采用新增两张表的方案，避免把定向问答与自由 QA 混在同一张表中。

### 4.1 directed_questions

字段建议：

- `id`: UUID，主键
- `question_text`: Text，题干
- `description`: Text，可空，补充说明
- `status`: String，`active` 或 `inactive`
- `created_at`: Timestamp

用途：

- 存储管理员维护的定向题库
- 控制题目是否在前台可见

### 4.2 directed_answers

字段建议：

- `id`: UUID，主键
- `question_id`: UUID，关联 `directed_questions.id`
- `student_name`: String
- `student_id`: String
- `nickname`: String
- `answer`: Text
- `status`: String，`visible` 或 `hidden`
- `is_adopted`: Boolean，默认 `false`
- `created_at`: Timestamp

用途：

- 存储学生对定向题目的回答
- 供前台展示和后台审核使用

### 4.3 约束与索引

- `directed_answers.question_id` 建索引
- `directed_answers.student_id` 建索引
- `directed_answers(question_id, student_id)` 建联合索引，支持后台采纳时快速取消旧记录
- 采纳唯一性不依赖单条唯一索引直接硬拦，而是在管理员采纳操作时执行“同题同学生先取消旧采纳，再采纳新答案”的事务化更新逻辑

## 5. 前台设计

### 5.1 首页结构

在现有首页 Tab 中新增一个独立 Tab：`定向问答`。

现有模块保留：

- QA 展示
- 排行榜
- 发布 QA

新增后：

- QA 展示
- 排行榜
- 发布 QA
- 定向问答

### 5.2 定向问答页面结构

页面拆为两个区域：

- 题目列表区
- 当前题目的回答区

题目列表区：

- 展示所有 `active` 状态题目
- 默认选中第一道题
- 支持点击切换当前题目

回答区：

- 顶部显示当前题目题干
- 中间为回答输入框
- 底部为当前题目的公开回答列表

### 5.3 学生提交流程

1. 页面加载时复用现有 `localStorage` 身份信息
2. 学生选择一个定向题目
3. 填写回答内容
4. 提交到 `directed_answers`
5. 提交成功后清空回答输入框
6. 刷新当前题目的回答列表

### 5.4 定向回答展示

每个题目下展示多条回答，单条回答显示：

- 昵称
- 回答内容
- 创建时间
- 已采纳标识

该展示结构与自由 QA 墙分离，不显示“问题 + 回答”卡片形式。

## 6. 后台设计

后台在现有管理页中新增两个管理区块，不新增单独路由。

### 6.1 题目管理

管理员可以：

- 新增题目
- 编辑题目文本和补充说明
- 启用题目
- 停用题目
- 删除题目

删除题目时需要同步删除其下回答，或在界面层阻止删除仅允许停用。为减少误操作，本次优先采用“允许停用，删除保留但需二次确认”的方式。

### 6.2 回答管理

管理员可以：

- 按题目查看回答
- 搜索昵称、学号、回答文本
- 采纳或取消采纳回答
- 隐藏或恢复显示回答
- 删除回答

### 6.3 采纳操作细则

采纳定向回答时：

1. 找出该回答对应的 `question_id` 和 `student_id`
2. 将该题下该学生其他 `is_adopted = true` 的记录改为 `false`
3. 将当前回答设为 `is_adopted = true`
4. 刷新后台列表

取消采纳时：

- 仅将当前记录的 `is_adopted` 改为 `false`

## 7. 排行榜设计

现有排行榜逻辑基于 `qa_records_public` 聚合自由 QA 的采纳数。

改造后：

- 继续查询自由 QA 的采纳数据
- 额外查询定向回答的采纳数据
- 以 `student_id` 为键合并两份统计
- 输出总采纳数和总提交数

如果一个学生同时参与两类内容：

- 总分为两边采纳数量之和
- 展示昵称优先使用最近可见记录中的昵称，或沿用当前简单策略，取聚合过程遇到的昵称

本次不额外拆分“自由 QA 得分”和“定向问答得分”的前台字段，避免增加 UI 复杂度。

## 8. 实现边界

### 8.1 代码组织

沿用当前项目风格，避免为本次功能新增过度抽象。

预计新增或修改：

- `src/pages/HomePage.vue`
- 新增 `src/pages/DirectedQA.vue`
- `src/pages/Leaderboard.vue`
- `src/pages/AdminView.vue`
- `supabase/migrations/...`

如有必要，可新增少量数据访问辅助函数，但不进行整站重构。

### 8.2 安全边界

当前后台本身缺乏真正鉴权。本次不顺手扩展为完整权限系统，否则会显著扩大需求范围。

本次目标是先把“定向问答”业务闭环实现完成。

## 9. 验证方案

项目当前没有现成测试框架，本次验证采用以下方式：

### 9.1 静态验证

- `vue-tsc -b`
- `vite build`

### 9.2 人工流程验证

至少验证以下流程：

1. 管理员创建定向题目
2. 前台能看到启用题目
3. 学生对同一题重复提交多条回答
4. 当前题目页能展示多条回答
5. 管理员采纳其中一条回答
6. 同题同学生旧采纳记录被自动取消
7. 排行榜分数正确增加
8. 本地站点可访问

## 10. 实施顺序

1. 增加数据库迁移
2. 实现前台定向问答页面
3. 扩展后台题目管理与回答管理
4. 改造排行榜聚合逻辑
5. 本地运行并手工验证

## 11. 已确认决策

- 定向问答是独立模块，不混入现有 QA 墙
- 同一学生可对同一题重复回答多次
- 同一学生在同一定向题下最多只能有 1 条已采纳回答
- 采纳新回答时，自动取消该学生该题下旧的已采纳回答
- 定向问答被采纳后并入现有总排行榜
